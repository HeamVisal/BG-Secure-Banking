\section{Abstract}

BG Bank is a secure banking web application built as a class project for Introduction to Parallel and Distributed Systems. The system demonstrates how a web client can communicate with separated backend services through Remote Procedure Call (RPC). The application uses Flask for the web interface, XML-RPC for distributed service communication, SQLite for storage, Fernet encrypted login tickets for session identity, hashed passwords for authentication, and a rule-based fraud detection module for transaction monitoring.

The main distributed design separates the application into a web layer, an authentication RPC service, a banking RPC service, and a database layer. This separation makes the project suitable for studying service boundaries, remote method calls, shared data access, and transaction workflows.

\section{Project Objectives}

The objectives of this project are:

\begin{itemize}
    \item To build a banking web application with registration, login, profile, account, transaction, and fraud report features.
    \item To apply XML-RPC as a distributed communication mechanism between the web application and backend services.
    \item To protect login sessions using encrypted tickets and protect sensitive transactions using an action password.
    \item To store users, customer profiles, accounts, transactions, and audit logs in a database.
    \item To detect suspicious transaction behavior using rule-based fraud scoring.
    \item To run the system in a containerized environment using Docker.
\end{itemize}

\section{System Architecture}

The system is divided into independent components. The browser communicates with the Flask web application. The Flask application does not directly perform all authentication and banking work; instead, it calls XML-RPC services. The authentication service handles registration and login, while the banking service handles balance, deposit, withdraw, transfer, history, profile, audit log, and fraud report operations. SQLite stores all persistent application data.

\section{RPC Service Design}

The project uses two XML-RPC services. The authentication service exposes registration and login methods. The banking service exposes methods for banking operations after the user has logged in.

\subsection{Authentication Service}

The authentication service validates user input during registration, hashes the login password and action password, creates the user profile and account, and returns structured RPC responses. During login, it verifies the password and returns an encrypted Fernet ticket.

\subsection{Banking Service}

The banking service receives the encrypted ticket from the Flask application. Before processing requests, it decrypts and validates the ticket to identify the current user. The service then performs account and transaction operations such as:

\begin{itemize}
    \item Getting dashboard summary and account details.
    \item Depositing money.
    \item Withdrawing money with action password verification.
    \item Transferring money to another user with action password verification.
    \item Viewing transaction history.
    \item Loading and trusting fraud reports.
    \item Reading and updating customer profile data.
\end{itemize}

\subsection{Transfer Workflow}

Transfer is the most complete distributed workflow because it uses the web app, banking RPC service, security module, database, and fraud detection module.

\begin{figure}[H]
    \centering
    \includegraphics[width=\textwidth]{RPC-Workflow-Diagram-GPT.png}
    \caption{XML-RPC transfer workflow}
\end{figure}

The transfer workflow is:

\begin{enumerate}
    \item The user submits receiver, amount, and action password in the Flask web app.
    \item Flask calls the banking XML-RPC method with the encrypted ticket and transfer data.
    \item The banking service decrypts the ticket and identifies the sender.
    \item The service validates the action password.
    \item Sender and receiver accounts are loaded from SQLite.
    \item The service checks balance and calculates new balances.
    \item Fraud detection checks transaction risk.
    \item Sender and receiver balances are updated.
    \item Transfer transaction records and audit logs are saved.
    \item The result is returned to Flask and displayed to the user.
\end{enumerate}

\section{Main Workflows}

\subsection{Registration and Login}

During registration, the user submits account and profile information. The authentication service validates the input, hashes the login password and action password, creates a user row, creates a customer profile, and creates a bank account. During login, the authentication service verifies the password and issues an encrypted ticket. Flask stores this ticket in the session and sends it to the banking service for later operations.

\subsection{Deposit}

Deposit requires a valid logged-in session but does not require the action password. The banking service validates the ticket, loads the user's account, calculates the new balance, runs fraud detection, updates the balance, records the transaction, and writes an audit log.

\subsection{Withdraw}

Withdraw requires the login ticket and action password. The service validates the ticket, checks the action password, verifies sufficient balance, calculates the new balance, runs fraud detection, updates the balance, records the transaction, and writes an audit log.

\subsection{Transfer}

Transfer requires the login ticket, receiver username, amount, and action password. The service prevents self-transfer, validates the action password, checks receiver account status, checks sender balance, calculates sender and receiver balances, runs fraud detection for the sender, updates both accounts, saves two transaction records, and writes audit logs for both users.

\subsection{Fraud Detection}

Fraud detection is rule-based. It calculates a risk score and marks transactions as suspicious when the score reaches the configured threshold. The rules include:

\begin{itemize}
    \item Account status problems.
    \item Large transaction amount.
    \item Amount much higher than the user's average transaction amount.
    \item Many transactions in a short time window.
    \item Large balance drop after withdraw or transfer.
    \item Transfers to many different receivers quickly.
    \item Transfer amount exceeding the daily transfer limit.
\end{itemize}

\section{Implementation Details}

\subsection{Encrypted Login Ticket}

After successful login, the authentication service creates a Fernet encrypted ticket containing the username, issue time, and expiry time. The banking service decrypts and validates this ticket before allowing banking operations. This approach separates authentication from banking operations while still allowing the banking service to identify the current user.

\subsection{Action Password}

The application uses a second password called the action password. It is required for sensitive money-moving operations such as withdraw and transfer. This gives stronger protection than using only the login password.

\subsection{Logging}

The project includes structured logging for the web app, authentication RPC service, banking RPC service, fraud detection module, and database layer. The logs help demonstrate how data moves across the distributed services during registration, login, deposit, withdraw, transfer, and fraud detection.

\section{Security Discussion}

The project includes several important security features:

\begin{itemize}
    \item Login passwords and action passwords are stored as hashes.
    \item Login tickets are encrypted and time-limited.
    \item Banking RPC methods validate the login ticket before processing.
    \item Withdraw and transfer require the action password.
    \item Fraud detection records suspicious transaction behavior.
    \item Audit logs record important account actions.
\end{itemize}

The project also has limitations that should be improved for production use:

\begin{itemize}
    \item Sensitive logging should be disabled because logs can expose password hashes, password fingerprints, encrypted tickets, and registration payloads.
    \item Deposit, withdraw, and transfer database writes should be made atomic so balances and transaction records cannot become inconsistent after an error.
    \item Web forms should include CSRF protection.
    \item Registration should use a database transaction so user, profile, and account creation roll back together on failure.
    \item Money values should use integer cents or Decimal instead of floating point values.
    \item Logical database relationships should be enforced with foreign keys where appropriate.
\end{itemize}

\section{Limitations and Future Improvements}

Future improvements include adding transactional database writes, enabling CSRF protection, using safer money storage, enforcing foreign keys, adding automated tests, improving error handling, and separating development logging from production logging. The system could also be extended with admin review tools, stronger fraud models, account freezing, notification services, and HTTPS deployment.

\section{Conclusion}

BG Bank demonstrates a distributed banking application using Flask and XML-RPC services. The project separates web presentation, authentication, banking logic, fraud detection, and database storage into clear components. It shows how encrypted tickets and action passwords can protect banking operations, and how rule-based fraud detection can help identify suspicious transactions. The application is suitable for demonstrating RPC communication, service coordination, shared database access, and security concerns in distributed systems.
